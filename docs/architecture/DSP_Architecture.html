02 — Per-Pad Playback Engine

Two ways to read a sample buffer

Each pad owns an AudioBuffer voice. Per block it snapshots its atomic state into a coherent StretchSnapshot, then branches once to one of two readers. The choice is useStretcher() — true whenever stretchRatio ≠ 1.0 or pitchSemiTones ≠ 0. This is an either/or dispatch per block, not a serial chain: when a pad is at normal stretch with no pitch shift, SoundTouch is never touched.

① Repitch reader — processWithRepitching

Speed change & direction only — pitch tracks speed, like varispeed tape. The JUCE interpolator reads the source straight to output.

source buffer › JUCE interp / resample › output SoundTouch bypassed
  • Per-output-sample step = speed × (fileSR / hostSR)
  • 4-point Lagrange / linear: dst = src[p1] + frac·(src[p2]−src[p1])
  • Negative speed ⇒ reverse; bulk copyFrom fast-path when speed=1 & rates match

② Time-stretch reader — processWithTimeStretch

Independent tempo & pitch. The same JUCE interpolator reads the source into an input scratch, and SoundTouch WSOLA warps that — two nested stages:

source buffer › JUCE interp → stretchInScratch › SoundTouch WSOLA › output
The JUCE read stage is fillInputScratch (bulk copy · LagrangeInterpolator · per-sample fallback); SoundTouch does the time/pitch warp.
  • Windows: SEQUENCE 60 ms · SEEKWINDOW 25 ms · OVERLAP 12 ms
  • Pitch active ⇒ speed via tempo; else via rate (keeps them independent)
  • Source read does not apply the stretch ratio — SoundTouch does, avoiding double-application
  • Feed/drain loop primes latency, clamps output to ±1.0; reverse widens windows to 35/18

Key nuance: the JUCE LagrangeInterpolator (source reading / resampling) runs in both paths — writing to the output in repitch mode, and to SoundTouch's input scratch in stretch mode. SoundTouch (time/pitch warp) runs only in the stretch path. Both readers then share the slice engine (beat-slice random, arp sequence, slice-repeater), ping-pong (direction flips every musical half-period), and the 50 ms SmoothedValue ramps on speed, stretch, pitch and speed-magnitude.

ChannelStrip::processDSPChain — fixed order

After the playback reader fills the pad's scratch buffer, its ChannelStrip runs the insert chain below in a fixed order. Each stage is gated by an enable flag and driven by an EffectEnvelope that ramps parameters over musical bars. A NodeProbe inspects the signal after most stages to locate clipping.

Delay

Multi-tap fractional line; feedback high-cut IIR + tanh saturation; optional ping-pong & wow/flutter modulation of the read tap.

S&H Filters

Per-strip RNG latches a new random cutoff & Q at each musical division (1/16…1/4), driven by an accumulating phase.

Granular

Clouds-style: 2 s capture ring, up to 48 grains, octave-quantized pitch, Hann/rect texture blend, tanh soft-clip.

Reverb

JUCE dsp::Reverb, roomSize 0.80, 20 ms pre-delay, perceptual wet = wet^0.85, limiter fore & aft.

Chorus / Auto-pan

Chorus: LFO-modulated fractional delay. Auto-pan: constant-power sine panner, rate from musical division.

Boundary de-click

Final guard smoothing block-edge discontinuities before the buffer leaves the strip.

04 — Threading Model

Who runs what, and how audio crosses threads

The 8 pads render concurrently on a lock-free, allocation-free pool — the audio thread is itself a worker, giving 4-way parallelism. Merging (summation) happens back on the audio thread because it is not independent. Disk I/O never touches the audio thread.

Lock-free load hand-off

Decoded LoadedAudioData crosses loader→audio via a 32-slot AbstractFifo + atomic pointer swap with hazard-style retainers. The audio thread never frees memory or blocks.

Deferred SoundTouch reset

Param changes set stretcherNeedsReset; the actual reset runs on the audio thread, so the message thread never touches the stretcher.

Prepared gate

An acquire-load atomic resourcesPrepared forces silence during AU release-time callbacks and bus reconfiguration (10-block grace).

05 — The Modifier System

How the graph rewrites itself in musical time

A "modifier" is a scheduled, probabilistically-selected command — not a DSP block itself. The ModifierScheduler fires one at each musical boundary; AppState::modifierTriggered switches on its type and mutates parameters on the targeted pads' AudioBuffer or ChannelStrip. The DSP above then renders the new configuration.

06 — De-Click / De-Pop Arsenal

Seven mechanisms that keep transitions silent

Glitch performance means constant discontinuities — slice jumps, engine switches, stretcher resets. Each is masked by a purpose-built equal-power crossfade or fade. All use fadeIn = sin(θ), fadeOut = cos(θ), θ = progress·π/2.

A · Slice crossfade

Equal-power blend on every slice / loop / file-boundary jump in the repitch path. Default 20 ms (882 samples @44.1k); length captured at start so mid-fade changes can't corrupt it.

B · Input-side crossfade

In stretch mode the discontinuity is blended into SoundTouch's input (≥50 ms) so the OLA engine never sees a hard edge.

C · Lookahead pre-prime

When slicing + stretching, the next slice is ramped into the stretcher input within its latency window, then continued past the boundary.

D · Transition fade

Multi-block equal-power fade from the stretchOutputRing tail when switching repitch↔stretch or after a reset. 20–80 ms, spans as many blocks as needed.

E · Startup fade-in

First block after a stretcher reset is pre-primed with ~2× latency and fades in linearly over 20–40 ms to mask the settling transient.

F · Underrun tail fade

If SoundTouch underfills a block, the tail fades to zero over ~2 ms instead of holding the last sample (which caused DC offset).

G · Output-side crossfade is a legacy fallback now rarely hit. Pad (−12 dB) and master gains are flat multiplies; smoothing is handled entirely by the parameter ramps and the crossfades above.

07 — Buffers & Data Structures

Where the samples actually live

LoadedAudioData

Ref-counted decoded source sample (stereo AudioBuffer<float> + sample rate + name). Read lock-free via an atomic pointer with retainer rotation; survives release/re-prepare so audio isn't reloaded on bus changes.

Pending-load FIFO

32-slot SPSC AbstractFifo. Loader thread writes decoded buffers; audio thread drains them and applies the loop window. Overflow is logged, not blocked.

stretchOutputRing

Stereo 100 ms circular buffer written every block. Feeds transition fades and snapshots so the previous engine's tail can crossfade into the new one.

Scratch buffers

perBufferScratch[8] + main scratchBuffer, all pre-allocated in prepareToPlay. Runtime setSize is guarded by jassertfalse — no allocation on the audio thread, ever.

Output topology: one stereo Mix bus plus 8 stereo per-pad buses (Ch1…Ch8) for multi-out routing in the host.
